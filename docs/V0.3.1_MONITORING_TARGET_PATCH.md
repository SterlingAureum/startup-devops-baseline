# v0.3.1 Monitoring Target Patch

## Problem

After v0.3.1, `demo-api` uses Argo Rollouts with ingress-nginx traffic routing.

The service model changed from:

```text
service/demo-api
```

to:

```text
service/demo-api-stable
service/demo-api-canary
```

However, the lightweight Prometheus configuration still scraped the old service DNS name:

```text
demo-api.startup-apps.svc.cluster.local:80
```

This caused the Prometheus target to become `down` with an error like:

```text
lookup demo-api.startup-apps.svc.cluster.local: no such host
```

## Fix

Update the Prometheus scrape target to the stable service:

```text
demo-api-stable.startup-apps.svc.cluster.local:80
```

The stable service is the correct default scrape target after the rollout has completed, and it represents the production/stable workload path.

## Refresh and reload

For a fresh GitOps deployment, no manual reload is required because Prometheus starts with the updated ConfigMap.

For an existing running cluster, restart or reload Prometheus after the ConfigMap is updated.

```bash
./scripts/restart-prometheus.sh
```

If restart does not take effect, restart Prometheus manually:

```bash
kubectl -n monitoring rollout restart deployment/prometheus
kubectl -n monitoring rollout status deployment/prometheus --timeout=180s
```

## Verify

```bash

```

Expected target:

```text

```

Expected health:

```text

```

Then run:

```bash
./scripts/validate.sh
```

The previous Prometheus warning should disappear after Prometheus has scraped the updated target.
