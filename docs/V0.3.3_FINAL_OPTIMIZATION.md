# v0.3.3 Final Optimization

## Purpose

This patch finalizes the v0.3.3 Prometheus canary analysis baseline.

It adds production-oriented notes and a small rollout safety setting:

```yaml
maxSurge: 1
maxUnavailable: 0
```

This keeps rollout resource overhead bounded during canary updates.

## Why this matters

During a canary rollout, old and new versions can exist at the same time.

For small demo workloads this is fine, but for production workloads it can be expensive:

```text
10 replicas x 20Gi memory = 200Gi normal capacity

If old 10 and new 10 coexist:
20 replicas x 20Gi memory = 400Gi peak capacity
```

For GPU workloads, the same issue can become much more expensive:

```text
stable: 10 GPU pods
canary: 10 GPU pods
peak: 20 GPUs
```

The local baseline should therefore demonstrate that rollout capacity is intentionally controlled.

## Apply

```bash
./scripts/apply-v0.3.3-final-rollout-settings.sh
```

Review:

```bash
git diff apps/demo-api/helm/values.yaml
```

Validate:

```bash
helm lint apps/demo-api/helm

helm template demo-api apps/demo-api/helm | grep -E "maxSurge|maxUnavailable|kind: AnalysisTemplate|demo-api-canary-health" -A10
```

Commit:

```bash
git add apps/demo-api/helm/values.yaml \
  apps/demo-api/helm/values-v0.3.3-final-optimization-snippet.yaml \
  scripts/apply-v0.3.3-final-rollout-settings.sh \
```

## Validation after Argo CD sync

```bash
kubectl get application -n argocd
kubectl -n startup-apps get rollout demo-api
kubectl -n startup-apps get analysistemplate

```

If you want to trigger another rollout, update the image tag to a new already-published GHCR image tag and commit the values change.

## Current v0.3.3 boundary

This version includes:

```text
- GHCR image publishing from v0.3.2
- Manual GitOps image promotion
- Argo Rollouts ingress-nginx canary routing
- Prometheus-based canary AnalysisTemplate
- Manual final promote
- Bounded rollout surge capacity
```

It does not yet include:

```text
- automatic values.yaml image tag update
- Argo CD Image Updater
- app/GitOps repository split
- real HTTP error-rate or latency based canary analysis
- EKS/ECR integration
```
