# v0.3.1 Ingress Canary Routing Plan

## Version name

```text
v0.3.1-ingress-canary-routing
```

## Goal

Extend the v0.3 Argo Rollouts baseline from replica-based canary to ingress-based canary traffic routing.

v0.3 demonstrates:

```text
Rollout state
ReplicaSet transition
manual promote
abort
rollback
```

v0.3.1 should demonstrate:

```text
Argo Rollouts
+ ingress-nginx traffic routing
+ real external HTTP canary weight control
```

## Why ingress traffic routing

For a startup or small-to-medium team, ingress-based canary is usually a better default than service mesh.

Reasons:

- ingress-nginx is already part of this repository
- fewer moving parts than Istio/Linkerd
- lower operational complexity
- easier to debug in local kind
- closer to the first production step for a small team

Service mesh can be documented later as an optional extension.

## Target architecture

```text
Client
  |
  v
ingress-nginx
  |
  +--> demo-api-stable Service
  |
  +--> demo-api-canary Service
          ^
          |
    Argo Rollouts controls canary weight
```

## Planned changes

### 1. Helm values

Add traffic routing config:

```yaml
rollout:
  enabled: true
  strategy:
    canary:
      enabled: true
      stableService: demo-api-stable
      canaryService: demo-api-canary
      trafficRouting:
        nginx:
          enabled: true
          stableIngress: demo-api
      steps:
        - setWeight: 20
        - pause: {}
        - setWeight: 50
        - pause:
            duration: 30s
        - setWeight: 100
```

### 2. Services

Replace the single service-only model with:

```text
service/demo-api-stable
service/demo-api-canary
```

Optionally keep:

```text
service/demo-api
```

as a compatibility alias only if it does not conflict with Rollouts traffic routing.

### 3. Ingress

Keep stable ingress:

```text
ingress/demo-api
```

Argo Rollouts will create or patch canary behavior depending on the NGINX traffic routing configuration.

### 4. Rollout template

Update Rollout strategy:

```yaml
strategy:
  canary:
    stableService: demo-api-stable
    canaryService: demo-api-canary
    trafficRouting:
      nginx:
        stableIngress: demo-api
    steps:
      - setWeight: 20
      - pause: {}
      - setWeight: 50
      - pause:
          duration: 30s
      - setWeight: 100
```

## Validation flow

### Check resources

```bash
kubectl get rollout -n startup-apps
kubectl get svc -n startup-apps
kubectl get ingress -n startup-apps
kubectl argo rollouts get rollout demo-api -n startup-apps
```

### Trigger a rollout

Update image tag or application version:

```text
0.1.0 -> 0.1.1
```

Then sync via Argo CD.

### Watch rollout

```bash
./scripts/rollout-watch.sh
```

### Promote rollout

```bash
./scripts/rollout-promote.sh
```

### Abort rollout

```bash
./scripts/rollout-abort.sh
```

## Important production note

In v0.3, `setWeight` mainly demonstrates replica-based progressive delivery.

In v0.3.1, NGINX traffic routing makes the canary closer to real external HTTP traffic splitting.

However, this is still an ingress-level canary. It is not service-to-service traffic governance. For deep east-west traffic control, mTLS, retries, circuit breaking, and service identity, a service mesh should be considered later as an optional extension.

## Acceptance criteria

v0.3.1 is complete when:

- demo-api Rollout uses NGINX traffic routing
- stable and canary Services exist
- stable Ingress exists
- canary weight changes are reflected in ingress-nginx behavior
- manual promote works
- abort/rollback works
- validation script passes
- docs explain ingress canary vs service mesh
