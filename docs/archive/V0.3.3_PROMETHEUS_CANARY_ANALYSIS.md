# v0.3.3 Prometheus Canary Analysis

## Goal

v0.3.3 adds a Prometheus-based analysis gate to the Argo Rollouts canary workflow.

The goal is to move from:

```text
manual observation + manual promote
```

to:

```text
canary traffic
  -> Prometheus analysis
  -> continue or fail the rollout
```

Argo Rollouts supports AnalysisTemplates and AnalysisRuns as progressive delivery checks. The Prometheus provider allows a rollout to query Prometheus during a canary step and use the result to decide whether the rollout should continue or fail.

## Scope

This version intentionally uses a small and reliable first analysis gate:

```text
up{job="demo-api-canary"} >= 1
```

This verifies that Prometheus can scrape the canary service during a rollout.

This is not yet a full business success-rate or error-rate gate. The current demo-api metrics do not expose HTTP status-code labels, so a true 5xx/error-rate analysis should be added in a later version after the app metrics are expanded.

## What changes

### 1. Prometheus scrape targets

The Prometheus config now scrapes both:

```text
demo-api-stable.startup-apps.svc.cluster.local:80
demo-api-canary.startup-apps.svc.cluster.local:80
```

This allows Rollout analysis to query canary-specific Prometheus targets.

### 2. AnalysisTemplate

A new Helm template creates:

```text
AnalysisTemplate/demo-api-canary-health
```

The metric query is:

```promql
sum(up{job="demo-api-canary"})
```

The success condition is:

```text
result[0] >= 1
```

### 3. Rollout steps

The canary flow becomes:

```text
20%
  wait 60s
  Prometheus analysis: canary target up
50%
  manual pause
100%
```

The 60-second pause gives Prometheus enough time to scrape the canary target before the AnalysisRun starts.

## Apply

```bash
./scripts/enable-demo-api-analysis.sh
```

Review the changes:

```bash
git diff apps/demo-api/helm/values.yaml
helm lint apps/demo-api/helm
helm template demo-api apps/demo-api/helm | grep -E "kind: AnalysisTemplate|analysis:|demo-api-canary-health" -A10
```

Commit and push:

```bash
git add apps/demo-api/helm/templates/analysis-template.yaml \
  apps/demo-api/helm/values.yaml \
  platform/monitoring/prometheus/configmap.yaml \
  scripts/enable-demo-api-analysis.sh \
  scripts/check-rollout-analysis.sh \
  scripts/generate-demo-api-traffic.sh \
  docs/V0.3.3_PROMETHEUS_CANARY_ANALYSIS.md \
  docs/V0.3.3_UPDATE_NOTES.md

git commit -m "feat: add prometheus canary analysis"
git push
```

## Runtime validation

After Argo CD sync:

```bash
kubectl get analysistemplate -n startup-apps
kubectl get rollout -n startup-apps
kubectl get application -n argocd
```

Restart Prometheus if the ConfigMap changed in an existing cluster:

```bash
kubectl -n monitoring rollout restart deployment/prometheus
kubectl -n monitoring rollout status deployment/prometheus --timeout=180s
```

Check targets:

```bash
kubectl -n monitoring port-forward svc/prometheus 19090:9090
curl -G "http://127.0.0.1:19090/api/v1/targets" | grep "demo-api"
```

Expected jobs:

```text
demo-api-stable
demo-api-canary
```

## Trigger a canary rollout

Build/publish a new image using the v0.3.2 GHCR workflow, update the Helm image tag, and push the values change.

Then watch:

```bash
./scripts/rollout-watch.sh
```

Generate traffic during the 20% step:

```bash
./scripts/generate-demo-api-traffic.sh
```

Check AnalysisRuns:

```bash
./scripts/check-rollout-analysis.sh
```

## Expected behavior

If Prometheus can scrape the canary service:

```text
AnalysisRun Successful
Rollout continues to the next step
```

If Prometheus cannot scrape the canary service:

```text
AnalysisRun Failed
Rollout becomes degraded or stops progressing
```

## Production note

This version is a safe first analysis gate.

A more production-like canary analysis should later add:

- HTTP status-code labels
- error-rate checks
- latency checks
- stable vs canary comparison
- automatic abort thresholds
