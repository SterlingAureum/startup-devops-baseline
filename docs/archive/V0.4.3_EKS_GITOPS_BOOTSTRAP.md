# v0.4.3 EKS GitOps Bootstrap

## Scope

This version connects the v0.4.2 EKS infrastructure baseline to the repository's GitOps workflow.

It adds:

- an IRSA role and IAM policy for AWS Load Balancer Controller;
- an EKS bootstrap script for the controller ServiceAccount and Argo CD;
- an `aws-dev` Argo CD root application;
- AWS Load Balancer Controller installation through an Argo CD Helm Application;
- a dedicated `values-aws-dev.yaml` for demo-api;
- an internet-facing ALB Ingress;
- an end-to-end AWS validation script.

## Environment boundary

The local environment keeps the Argo Rollouts and ingress-nginx canary implementation. The AWS development environment intentionally uses a standard Deployment, ClusterIP Service, and ALB Ingress.

This avoids coupling the first EKS GitOps validation to ALB-based progressive delivery. ALB traffic routing for Argo Rollouts can be added after the AWS baseline is stable.

## Bootstrap boundary

Terraform creates the IAM role for AWS Load Balancer Controller. The bootstrap script reads the role ARN from Terraform output and annotates the `kube-system/aws-load-balancer-controller` ServiceAccount.

Argo CD then owns the controller Helm release and the demo-api application. The ServiceAccount annotation remains a bootstrap dependency because the AWS account ID and role ARN must not be hardcoded in Git.

## Deployment

```bash
terraform -chdir=infra/terraform/aws/environments/dev apply

./scripts/bootstrap-eks-argocd.sh

REPO_URL=https://github.com/<owner>/startup-devops-baseline.git \
TARGET_REVISION=feature/v0.4-aws-eks-baseline \
./scripts/deploy-aws-dev-root-app.sh

./scripts/validate-aws-dev.sh
```

## Expected flow

```text
Terraform
  -> EKS and IAM
  -> annotated controller ServiceAccount
  -> Argo CD installation
  -> aws-dev root Application
  -> AWS Load Balancer Controller
  -> demo-api Deployment and Service
  -> internet-facing ALB
```

## Costs

The ALB adds hourly and LCU-based charges while it exists. EKS, EC2 nodes, NAT Gateway, EBS volumes, public IPv4 addresses, and data transfer may also incur charges.

Use the cleanup script before destroying the infrastructure:

```bash
./scripts/cleanup-aws-dev.sh
terraform -chdir=infra/terraform/aws/environments/dev destroy
```

The script deletes the root Application and waits until the demo-api Ingress is gone. Confirm that the ALB has also disappeared from AWS before destroying the VPC.
