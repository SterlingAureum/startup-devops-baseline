# v0.5.1 Karpenter GitOps Installation

## Objective

Install the Karpenter 1.14.0 CRDs and controller through Argo CD while keeping
node provisioning disabled until the repository defines an `EC2NodeClass` and
`NodePool`.

## Scope

This increment adds:

- a dedicated `karpenter-crd` Argo CD Application;
- a Karpenter controller Argo CD Application with Helm CRD installation
  disabled;
- an IRSA-annotated `karpenter` ServiceAccount created during bootstrap;
- controller scheduling on the two stable `workload=system` Managed Node Group
  nodes;
- controller and CRD validation;
- Terraform-driven rendering of the AWS Load Balancer Controller VPC ID.

This increment does not add:

- `EC2NodeClass`;
- `NodePool`;
- Spot or On-Demand workload capacity;
- consolidation behavior;
- scheduling migration for application workloads.

## GitOps Ownership

The aws-dev root Application recursively manages the platform directory and
preserves one environment-specific field:

```yaml
ignoreDifferences:
  - group: argoproj.io
    kind: Application
    name: aws-load-balancer-controller
    namespace: argocd
    jsonPointers:
      - /spec/source/helm/valuesObject/vpcId
syncPolicy:
  syncOptions:
    - RespectIgnoreDifferences=true
```

The ALB Application file is a template containing `__VPC_ID__`. The bootstrap
script reads `vpc_id` from Terraform, renders a temporary Application manifest,
and applies it after Argo CD becomes ready but before the root Application
syncs. The root continues to own and prune the child Application while
preserving only its live VPC ID; all other Helm values stay Git-managed. This
prevents a real environment ID from being committed and prevents the root
Application from restoring the template marker over the rendered live object.

Karpenter uses normal GitOps ownership:

```text
aws-dev root Application
├── karpenter-crd Application
├── karpenter Application
└── demo-api Application
```

The CRD Application uses sync wave `0`; the controller Application uses sync
wave `5` and `skipCrds: true`.

## Installation

Run from the repository root:

```bash
./scripts/bootstrap-eks-argocd.sh
./scripts/deploy-aws-dev-root-app.sh
```

The branch defaults to:

```text
feature/v0.5-karpenter-autoscaling
```

## Validation

Run:

```bash
./scripts/validate-karpenter-controller.sh
```

The validation checks:

- the two Karpenter Argo CD Applications;
- the NodePool, NodeClaim, and EC2NodeClass CRDs;
- the Karpenter ServiceAccount IRSA annotation;
- the Karpenter Deployment rollout;
- placement of running controller pods on `workload=system` nodes;
- the current NodePool and NodeClaim state.

An empty NodePool and NodeClaim listing is the expected v0.5.1 result. It proves
that the controller is installed without authorizing a node provisioning
policy.

## Acceptance Criteria

- `bootstrap-eks-argocd.sh` succeeds without a manually committed VPC ID.
- `karpenter-crd` and `karpenter` Applications exist.
- the three required CRDs exist.
- the Karpenter Deployment is available.
- controller pods run on stable system nodes.
- no `EC2NodeClass`, `NodePool`, or dynamically provisioned Karpenter node is
  introduced by this increment.
