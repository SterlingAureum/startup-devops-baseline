# v0.5.2 Karpenter EC2NodeClass

## Objective

Add the AWS launch and discovery configuration that future Karpenter NodePools
will reference, while preserving the no-node-provisioning boundary.

## Resource

The GitOps-managed resource is:

```text
apiVersion: karpenter.k8s.aws/v1
kind: EC2NodeClass
name: application
```

It configures:

- the Terraform-managed Karpenter node IAM role;
- EKS-optimized Amazon Linux 2023 AMI discovery;
- private-subnet discovery through `karpenter.sh/discovery`;
- EKS cluster security-group discovery through the same tag;
- required IMDSv2 with hop limit `1`;
- an encrypted 30 GiB gp3 root volume;
- no public IPv4 address;
- project, environment, and ownership tags.

## IAM Role Contract

`EC2NodeClass.spec.role` requires an IAM role name rather than an ARN. The
Terraform module creates the deterministic name:

```text
startup-devops-baseline-dev-karpenter-node-role
```

`validate-karpenter-nodeclass.sh` compares the manifest value with Terraform
output `karpenter_node_role_name`. Recreating the IAM role with the same
Terraform project and environment therefore does not require editing Git.

## AMI Policy

The aws-dev environment uses:

```text
al2023@latest
```

This allows the development environment to validate current EKS 1.36 AL2023
AMI discovery. A production environment should replace `latest` with a tested
and explicitly promoted AL2023 release before it introduces a NodePool.

## Sync Ordering

The resource uses Argo CD sync wave `10`. The Karpenter CRD Application is wave
`0`, and the controller Application is wave `5`.

`SkipDryRunOnMissingResource=true` prevents the initial root-Application dry
run from failing before the EC2NodeClass CRD has been registered.

## Validation

After pushing the v0.5 branch, run:

```bash
./scripts/deploy-aws-dev-root-app.sh
./scripts/validate-karpenter-nodeclass.sh
```

The validation checks:

- `Ready=True`;
- the EC2NodeClass role matches the Terraform output;
- at least two Terraform private subnets were discovered;
- the Terraform EKS cluster security group was discovered;
- AMIs and an instance profile were discovered;
- no NodePool exists;
- no NodeClaim exists;
- no Karpenter-provisioned node exists.

## Acceptance Boundary

Creating an EC2NodeClass does not request capacity. Karpenter only gains a
launch configuration in this increment. A later NodePool defines which
workloads, instance families, architectures, capacity types, and disruption
policies may use that configuration.

The controller does create a generated IAM instance profile for the node role.
The destroy and cleanup scripts therefore delete EC2NodeClasses and wait for
their finalizers before removing the controller or running Terraform destroy.
