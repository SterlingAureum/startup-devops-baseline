# v0.5.3 Karpenter On-Demand NodePool

## Scope

This increment turns the v0.5.2 launch template and discovery configuration
into controlled application capacity:

- one GitOps-managed `application-ondemand` NodePool;
- explicit workload isolation;
- bounded On-Demand instance selection;
- idle validation without provisioning;
- a separate scale-out and scale-in smoke test;
- dependency-aware cleanup and destroy handling.

## Scheduling Contract

The NodePool labels every node with:

```yaml
workload: application
capacity-owner: karpenter
```

It also applies:

```yaml
dedicated=application:NoSchedule
```

Application pods must therefore select `workload=application` and explicitly
tolerate the dedicated taint. Platform controllers remain on the stable
Managed Node Group labeled `workload=system`.

## Capacity Guardrails

The development NodePool permits:

- Linux `amd64`;
- On-Demand capacity only;
- generation 6 or newer `c`, `m`, and `r` instance categories;
- 2 or 4 vCPUs per instance;
- at most 4 total vCPUs, 16 GiB memory, and 2 nodes.

Empty or underutilized nodes become consolidation candidates after one minute.
Nodes expire after seven days.

## Validation

The normal validation path checks the NodePool configuration and requires an
idle Karpenter baseline:

```bash
./scripts/validate-all.sh
```

It does not create an EC2 node.

The controlled smoke test creates a temporary workload, validates its
NodeClaim and On-Demand node, removes the workload, and waits for scale-in:

```bash
./scripts/run-karpenter-scale-test.sh
```

The smoke test can incur a small temporary EC2 and EBS charge. It requires no
pre-existing NodeClaims or Karpenter nodes so its cleanup scope is unambiguous.

## Development AMI Policy

The EC2NodeClass continues to follow `al2023@latest` for development
validation. A production environment should promote an explicitly tested AMI
alias rather than following the latest release automatically.

## Next Increment

The next Karpenter increment can introduce Spot capacity and validate the
existing interruption queue and EventBridge foundation against a
Spot-compatible workload policy.
