# v0.5.4 Karpenter Spot NodePool

## Scope

This increment adds a second, isolated application capacity tier:

- one GitOps-managed `application-spot` NodePool;
- mutually exclusive On-Demand and Spot scheduling contracts;
- bounded Spot instance selection;
- runtime interruption-readiness validation;
- a controlled Spot scale-out and scale-in test;
- cleanup coverage for both smoke-test namespaces.

## Scheduling Contract

The On-Demand pool uses:

```yaml
capacity-tier: on-demand
dedicated=application:NoSchedule
```

The Spot pool uses:

```yaml
capacity-tier: spot
dedicated=application-spot:NoSchedule
```

A workload must tolerate the selected tier's taint. The two smoke workloads
also select the intended NodePool name directly, making the validation
deterministic.

## Spot Flexibility

The Spot pool permits:

- Linux `amd64`;
- Spot capacity only;
- generation 6 or newer `c`, `m`, and `r` categories;
- 2 or 4 vCPUs per instance;
- at most 4 total vCPUs, 16 GiB memory, and 2 nodes.

The category and generation constraints leave Karpenter multiple compatible
instance types instead of tying Spot provisioning to one market. The test does
not fall back to On-Demand when Spot capacity is unavailable.

## Interruption Readiness

The zero-capacity validation checks:

- `INTERRUPTION_QUEUE` on the running Karpenter controller;
- the Terraform-managed encrypted SQS queue;
- the enabled Spot interruption EventBridge rule;
- the EventBridge target ARN against the queue ARN.

Run:

```bash
./scripts/validate-all.sh
```

This confirms the configured event path without generating an interruption.

## Spot Capacity Test

Run intentionally:

```bash
./scripts/run-karpenter-spot-test.sh
```

The script:

1. validates interruption readiness;
2. creates a temporary Spot-only workload;
3. waits for a NodeClaim and node;
4. verifies Kubernetes capacity labels;
5. resolves the EC2 instance ID and confirms `InstanceLifecycle=spot`;
6. deletes the workload;
7. waits for consolidation-driven scale-in.

The test incurs a small temporary EC2 and EBS charge. Spot capacity is
best-effort and can be unavailable for the allowed instance markets.

## Interruption Test Boundary

This increment does not send a fabricated SQS event. A queue-only synthetic
message does not reproduce EC2's actual interruption lifecycle. The next
increment can use AWS Fault Injection Service to issue a real two-minute Spot
interruption notice and validate replacement, pod rescheduling, and cleanup.
